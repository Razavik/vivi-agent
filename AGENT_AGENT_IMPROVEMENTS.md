# Улучшения агента

Статусы:

- `[done]` - уже реализовано или в основном закрыто
- `[in_progress]` - частично есть, требует доведения
- `[planned]` - стоит сделать следующим
- `[later]` - полезно, но не первоочередно

## Цель

Сделать агента не просто исполнителем tool-call циклов, а надежным рабочим помощником, который:

- корректно понимает задачу и ограничения пользователя;
- планирует, выполняет и сам проверяет результат;
- безопасно работает с файлами, системой и внешними сервисами;
- умеет продолжать работу после ошибок и перезапуска;
- прозрачно показывает, что делает и почему.

## 1. Review-шаг перед финальным ответом

### Кейс

Агент завершает задачу и отвечает пользователю, но не всегда проверяет, что задача реально выполнена: файлы могли не измениться, тесты могли не запускаться, команда могла упасть, часть ограничений пользователя могла быть пропущена.

### Что сделать

- Добавить обязательный внутренний шаг `review_before_final`.
- Проверять исходную задачу против фактического результата.
- Проверять, были ли выполнены заявленные проверки.
- Проверять, не остались ли активные run, незавершенные tool calls или pending confirmations.
- В финальном ответе явно писать, что проверено, а что не удалось проверить.

### Acceptance criteria

- Агент не отправляет финальный ответ после кодовых изменений без попытки проверки.
- Если проверка невозможна, агент сообщает конкретную причину.
- Для multi-agent задач директор сверяет результаты саб-агентов перед ответом.

### Статус

`[planned]`

---

## 2. Quality gates по типу задачи

### Кейс

Разные задачи требуют разных минимальных проверок. Сейчас агент может ограничиться логическим выводом, хотя нужен build, typecheck, тесты или проверка файла.

### Что сделать

Ввести таблицу quality gates:

| Тип задачи | Минимальная проверка | Статус |
| --- | --- | --- |
| Python backend | `compileall`, unit tests, smoke import | `[planned]` |
| React frontend | `tsc`, build, визуальная проверка при UI-изменениях | `[planned]` |
| Markdown/docs | проверка UTF-8, открытие/чтение итогового файла | `[planned]` |
| Работа с файлами | existence check, размер, кодировка UTF-8 | `[planned]` |
| Системные команды | exit code, stdout/stderr summary, rollback notes | `[planned]` |
| Multi-agent run | статусы всех run, ошибки, артефакты | `[planned]` |

### Acceptance criteria

- Агент выбирает quality gate автоматически по типу изменений.
- Результат проверки сохраняется в run metadata.
- Финальный ответ содержит краткую строку `Проверка: ...`.

### Статус

`[planned]`

## 3. Структурированный результат саб-агентов

### Кейс

Саб-агенты могут возвращать свободный текст. Директору сложно надежно понять: задача выполнена, заблокирована, какие файлы изменены, чем подтвержден результат.

### Что сделать

Заставить саб-агентов возвращать результат по схеме:

```json
{
  "status": "done|blocked|failed",
  "summary": "Короткий итог",
  "changed_files": [],
  "created_artifacts": [],
  "verification": [],
  "risks": [],
  "needs_user_input": false,
  "question": ""
}
```

### Acceptance criteria

- `delegate_task` и `delegate_parallel` возвращают структурированный результат.
- Директор не считает задачу выполненной, если `status != done`.
- UI показывает `verification`, `risks`, `changed_files` отдельно от summary.

### Статус

`[in_progress]`

## 4. Контракты делегирования

### Кейс

Если директор делегирует задачу слишком общо, саб-агент может трогать лишние файлы, дублировать работу другого агента или вернуть непроверяемый ответ.

### Что сделать

Каждая делегация должна содержать:

- цель;
- контекст;
- allowed files / forbidden files;
- ожидаемый результат;
- обязательные проверки;
- формат ответа;
- критерий остановки;
- что делать при блокере.

### Пример контракта

```json
{
  "goal": "Проверить backend run lifecycle",
  "scope": ["src/agent", "src/web"],
  "do_not_touch": ["client"],
  "expected_output": "Список рисков и точек исправления",
  "verification": ["Найти места создания, обновления и завершения run"],
  "return_schema": "SubAgentResult"
}
```

### Acceptance criteria

- `DelegateTools` нормализует задачу в контракт.
- Саб-агент видит ограничения в prompt payload.
- Директор проверяет, что результат соответствует контракту.

### Статус

`[planned]`

## 5. Улучшенная рабочая память

### Кейс

История чата полезна, но она быстро превращается в шум. Агенту нужна компактная память: предпочтения пользователя, особенности проекта, прошлые ошибки, команды запуска.

### Что сделать

Разделить память на блоки:

- `user_preferences`;
- `project_facts`;
- `known_environment_issues`;
- `successful_commands`;
- `failed_attempts`;
- `decision_log`;
- `agent_self_notes`.

Добавить:

- `schema_version`;
- лимиты хранения;
- summary вместо полной истории;
- TTL для временных фактов;
- ручную очистку конкретного блока.

### Acceptance criteria

- Агент не тащит весь event log в prompt.
- Память можно посмотреть и очистить по разделам.
- Факты из памяти имеют источник и дату обновления.

### Статус

`[in_progress]`

---

## 6. Более точные статусы run

### Кейс

Статусы `running`, `paused`, `finished`, `failed` недостаточны. Агенту важно отличать ожидание пользователя от ожидания зависимости или внутреннего review.

### Что сделать

Добавить статусы:

- `queued`;
- `running`;
- `blocked`;
- `waiting_user`;
- `waiting_dependency`;
- `waiting_tool`;
- `reviewing`;
- `cancelling`;
- `cancelled`;
- `failed`;
- `done`.

### Acceptance criteria

- RunRegistry принимает только допустимые статусы.
- UI отображает статус явно.
- Supervisor умеет реагировать на `blocked`, `waiting_user`, `waiting_dependency`.

### Статус

`[in_progress]`

---

## 7. Replay/debug режим

### Кейс

Когда агент ошибается, трудно понять, где именно: prompt, ответ модели, tool call, результат инструмента или финальное решение директора.

### Что сделать

Для каждого run сохранять:

- входной user task;
- system prompt version;
- messages payload;
- LLM raw response;
- parsed action;
- tool call args;
- tool result;
- event timeline;
- final review result.

### Acceptance criteria

- Есть API `GET /api/runs/{run_id}/replay`.
- Есть UI-страница timeline/replay.
- Можно экспортировать replay в JSON.

### Статус

`[planned]`

---

## 8. Классификация ошибок и retry policy

### Кейс

Не все ошибки одинаковые. Timeout LLM, неверный JSON, ошибка аргументов инструмента и отказ доступа требуют разных действий.

### Что сделать

Ввести типы ошибок:

- `llm_timeout`;
- `llm_invalid_json`;
- `tool_validation_error`;
- `tool_runtime_error`;
- `permission_denied`;
- `network_error`;
- `user_cancelled`;
- `policy_blocked`;
- `unknown_error`.

Для каждого типа определить:

- retry allowed;
- max retries;
- backoff;
- нужна ли помощь пользователя;
- что писать в memory.

### Acceptance criteria

- Ошибки попадают в crash diagnostics с типом.
- Агент не повторяет бессмысленно одинаковый tool call.
- Пользователь получает конкретную причину, а не общий `ошибка инструмента`.

### Статус

`[planned]`

---

## 9. Safety policy для опасных действий

### Кейс

Подтверждение опасного действия полезно, но пользователю нужно понимать последствия: какие пути затронуты, можно ли откатить, почему это действие нужно.

### Что сделать

Перед опасными действиями показывать:

- action;
- risk level;
- affected paths/resources;
- reason;
- rollback possibility;
- safer alternative;
- exact command or operation.

### Acceptance criteria

- Destructive/file/system/network действия имеют объяснимое подтверждение.
- PathGuard применяется перед отображением подтверждения.
- Подтверждения сохраняются при рестарте.

### Статус

`[in_progress]`

---

## 10. Метрики качества агента

### Кейс

Нужно понимать, становится ли агент лучше: быстрее ли решает задачи, меньше ли ошибается, чаще ли проверяет результат.

### Что сделать

Собирать метрики:

- task duration;
- number of tool calls;
- number of failed tool calls;
- retry count;
- review pass/fail;
- tests attempted;
- tests passed;
- user intervention count;
- average context tokens;
- final answer completeness score.

### Acceptance criteria

- Метрики доступны в run details.
- Есть aggregate dashboard.
- Можно сравнить последние N задач.

### Статус

`[later]`

## Минимальный первый пакет работ

1. Добавить `SubAgentResult` / `DirectorResult` схемы.
2. Обновить `delegate_task` и `delegate_parallel`, чтобы они возвращали структурированные результаты.
3. Добавить `review_before_final` в цикл директора.
4. Добавить quality gates для backend/frontend/docs.
5. Добавить статусы `blocked`, `waiting_user`, `reviewing`.
6. Сохранять `verification` и `risks` в `RunRegistry`.
7. Обновить UI: показывать статус, проверки, риски и измененные файлы.

## Что считать успехом

Агент считается улучшенным, если после выполнения задачи он может ответить на три вопроса:

1. Что именно было сделано?
2. Чем это подтверждено?
3. Какие риски или непроверенные места остались?
