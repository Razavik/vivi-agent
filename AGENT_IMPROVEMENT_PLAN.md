# Полный план доработок агента

## Цель

Перевести проект из состояния "MVP-оркестратор с разовыми делегациями" в устойчивую multi-agent систему с:

- надёжной памятью
- корректной параллельной работой
- постоянным supervision loop у директора
- двусторонним real-time взаимодействием между директором и саб-агентами
- корректным UI для одновременных запусков
- наблюдаемостью, тестами и безопасностью

---

## 1. Текущее состояние и ключевые проблемы

### 1.1 Архитектурно уже есть

- директор и специализированные саб-агенты `file`, `system`, `web`, `telegram`
- реестр агентов и реестр инструментов
- event streaming в UI
- базовая память
- базовая валидация действий
- делегирование одной задачи и параллельного набора задач

### 1.2 Главные проблемы

- идентичность запуска завязана на `agent_name`, а не на отдельный `run_id`
- параллельные вызовы одного и того же саб-агента конфликтуют
- память небезопасна при многопоточном доступе
- директор пока не является постоянным supervisor-агентом
- нет двустороннего operational-канала директор <-> саб-агент
- UI отображает агент-центричную, а не run-центричную модель
- слабая валидация аргументов инструментов
- нет нормального recovery после сбоев
- нет проектных тестов на оркестрацию

---

## 2. Полный список направлений доработки

### 2.1 Базовая архитектура и идентичность запусков

#### Цель

Сделать так, чтобы каждая работа саб-агента была отдельной сущностью, а не просто "очередным вызовом агента по имени".

#### Задачи

- ввести `run_id` для каждого запуска саб-агента
- ввести `session_id` для сессии директора
- все события, статусы, сообщения, панели UI и результаты привязать к `run_id`
- перестать использовать `agent_name` как уникальный идентификатор конкретной работы
- ввести единый формат event envelope:
  - `event_id`
  - `timestamp`
  - `session_id`
  - `run_id`
  - `agent_name`
  - `event_type`
  - `payload`

#### Что исправит

- конфликт одинаковых запусков одного саб-агента
- перетирание результатов в параллельной делегации
- ошибочную корреляцию событий в UI

---

### 2.2 Исправление текущей параллельной делегации

#### Проблема

Сейчас `delegate_parallel` складывает результаты по `agent_name`, поэтому повторный запуск одного и того же агента перезаписывает предыдущий результат.

#### Задачи

- переделать `delegate_parallel` так, чтобы он возвращал список результатов, а не словарь по `agent_name`
- каждый элемент результата должен содержать:
  - `run_id`
  - `agent_name`
  - `status`
  - `success`
  - `result`
  - `steps`
- добавить поддержку нескольких одновременных запусков одного типа агента
- в UI перестать хранить одну панель на один `agent_name`

#### Дополнительно

- добавить лимиты на `max_workers`
- добавить обработку частичных падений
- сохранить порядок исходных задач или явно возвращать его через `index`

---

### 2.3 Потокобезопасная память и надёжное хранение состояния

#### Проблема

Сейчас память работает по модели `load -> modify -> write`, из-за чего возможны race condition и потеря данных при параллельной записи.

#### Этап 1. Быстрый защитный фикс

- добавить file lock на операции чтения/обновления памяти
- реализовать atomic write через временный файл и `os.replace`
- повторно читать актуальные данные уже после захвата lock
- добавить защиту от битого JSON

#### Этап 2. Перестройка модели памяти

- разделить:
  - память директора
  - память саб-агентов
  - историю конкретного run
  - event log
  - артефакты
- добавить `schema_version`
- добавить retention policy

#### Этап 3. Миграция на SQLite

- перейти с JSON-файлов на SQLite
- сделать транзакционную запись:
  - сообщений
  - событий
  - статусов run
  - артефактов
- подготовить recovery после рестарта процесса

#### Рекомендуемые сущности хранения

- `sessions`
- `runs`
- `messages`
- `events`
- `artifacts`
- `agent_memory`
- `state_snapshots`

---

### 2.4 RunRegistry и управление жизненным циклом запусков

#### Цель

Ввести центральный реестр всех активных и завершённых запусков.

#### Новые сущности

##### `RunContext`

- `run_id`
- `session_id`
- `agent_name`
- `status`
- `goal`
- `priority`
- `created_at`
- `updated_at`
- `current_step`
- `current_plan`
- `inbox`
- `outbox`
- `artifacts`
- `dependencies`
- `cancel_token`

##### `RunRegistry`

Должен уметь:

- зарегистрировать новый run
- получить run по `run_id`
- отдать список активных run
- обновить статус
- сохранить metadata
- завершить run
- отменить run
- пометить run как paused/resumed

#### API-поверхность

- `GET /runs`
- `GET /runs/{run_id}`
- `POST /runs/{run_id}/cancel`
- `POST /runs/{run_id}/pause`
- `POST /runs/{run_id}/resume`
- `POST /runs/{run_id}/message`
- `POST /runs/{run_id}/replace-task`

### 2.5 MessageBus и двустороннее общение директор <-> саб-агент

#### Цель

Сделать не просто лог событий, а рабочую шину команд и ответов.

#### Требуемые направления сообщений

- директор -> саб-агент
- саб-агент -> директор
- системные события -> директор
- backend -> UI

#### Типы сообщений

- `run_started`
- `progress_update`
- `heartbeat`
- `question_to_director`
- `answer_from_director`
- `directive`
- `pause_requested`
- `resume_requested`
- `cancel_requested`
- `task_replaced`
- `artifact_created`
- `dependency_wait`
- `dependency_ready`
- `run_finished`
- `run_failed`

#### Требования

- каждое сообщение должно иметь correlation id
- сообщения должны коррелироваться с `run_id`
- сообщения должны логироваться
- UI-стрим не должен быть единственным источником истины

---

### 2.6 Превращение саб-агентов в interruptible worker'ов

#### Цель

Саб-агенты должны уметь не только завершать задачу до конца, но и реагировать на внешнее управление.

#### Новое поведение

- регулярная проверка inbox между шагами
- поддержка `cancel`
- поддержка `pause`
- поддержка `resume`
- поддержка `replace_task`
- поддержка входящих уточняющих сообщений от директора
- поддержка heartbeat

#### Что нужно изменить

- добавить interruption points в цикл `SubAgent.run()`
- ввести статусную модель:
  - `queued`
  - `running`
  - `waiting_input`
  - `paused`
  - `cancelling`
  - `cancelled`
  - `finished`
  - `failed`
- реализовать мягкую отмену:
  - не рвать текущий tool step посередине без крайней необходимости
- реализовать корректную смену задачи:
  - либо в текущем run
  - либо через handoff в новый run

---

### 2.7 Эволюция директора в постоянного supervisor-агента

#### Цель

Сделать директора long-running оркестратором, который постоянно наблюдает за системой и вмешивается при необходимости.

#### Что должен уметь директор

- видеть все активные run
- получать их heartbeat
- получать вопросы от саб-агентов
- отправлять им команды
- отменять и ставить на паузу run
- менять приоритеты
- запускать новые run по результатам предыдущих
- реагировать на зависания, циклы и тупики

#### Нужные компоненты

##### `SupervisorLoop`

Цикл должен:

- читать новые события
- обновлять `WorldState`
- принимать решение о вмешательстве
- писать команды в inbox соответствующих run
- при необходимости отправлять промежуточные сообщения пользователю

##### `WorldState`

Должен содержать:

- активные run
- их статусы
- зависимости
- артефакты
- открытые вопросы
- текущие риски
- узкие места

#### Новые инструменты директора

- `start_sub_agent`
- `inspect_runs`
- `message_sub_agent`
- `pause_sub_agent`
- `resume_sub_agent`
- `cancel_sub_agent`
- `replace_sub_agent_task`
- `reprioritize_run`
- `wait_for_event`
- `summarize_system_state`

### 2.8 Handoff, зависимости и артефакты

#### Цель

Позволить run'ам работать как граф задач, а не только как набор изолированных вызовов.

#### Что добавить

- явную сущность `artifact`
- metadata артефактов:
  - `artifact_id`
  - `producer_run_id`
  - `type`
  - `location`
  - `summary`
- dependency graph между run

#### Примеры

- `web` собирает данные -> `file` формирует отчёт
- `system` делает screenshot -> `file` или `web` анализирует
- `file` находит конфиг -> `system` применяет

#### Новые механизмы

- `handoff_run`
- `wait_for_dependency`
- `mark_artifact_ready`
- `dependency_ready`

### 2.9 Улучшение валидатора, policy и безопасности

#### Проблемы

- сейчас валидируются в основном только обязательные аргументы
- risk model слишком грубая для long-running multi-agent системы

#### Что сделать

- проверка типов аргументов
- проверка структуры списков и вложенных объектов
- проверка допустимых enum-значений
- ограничение лишних полей
- введение run-level permissions

#### Новые уровни доступа

- read-only
- file-write
- system-control
- network-enabled
- destructive

#### Дополнительные policy-правила

- кто имеет право останавливать run
- какие действия требуют user confirmation
- какие действия требуют director approval
- квоты на:
  - число шагов
  - число tool invocations
  - runtime
  - внешние запросы

#### Детекторы проблем

- orchestration loop detection
- deadlock detection
- watchdog на зависшие run

### 2.10 Улучшение UI под реальную multi-agent оркестрацию

#### Проблема

UI сейчас агент-центричен и плохо подходит для нескольких параллельных запусков одного агента.

#### Что изменить

- отображать:
  - активные run
  - paused run

  - завершённые run
- показывать `agent_name` как тип worker'а, а не как идентификатор конкретного выполнения

#### Новые элементы UI

- список run'ов
- supervisor dashboard
- timeline событий
- inbox/outbox просмотры
- панель зависимостей и артефактов
- кнопки:
  - pause
  - resume
  - cancel
  - send message
  - replace task
  - reprioritize

#### Исправления текущей логики UI

- отказаться от матчирования по "последнему running pane"
- отказаться от корреляции вопросов/ответов по тексту вопроса
- все панели и события маппить по `run_id`

### 2.11 Обновление prompt-слоя и поведения моделей

#### Для директора

Переписать prompt так, чтобы директор:

- постоянно наблюдал за активными run

#### Для саб-агентов

### 2.13 Recovery и устойчивость к перезапуску

#### Цель

Поддержать долгоживущего директора и долгоживущие run.

#### Что сделать

- сохранять активные run в persistent storage
- сохранять inbox/outbox состояние
- сохранять pending confirmations
- сохранять snapshots `WorldState`
- при старте процесса:

  - переводить их в `aborted`, `recovering` или восстанавливать

### 2.14 Производительность и стоимость

#### Риски

Supervisor runtime будет дороже и тяжелее текущей модели.

#### Что оптимизировать

- не пересобирать полный контекст на каждый чих
- делать инкрементальные summaries
- хранить компактное состояние run
- не слать в модель весь event log
- вводить adaptive heartbeat
- ограничивать частоту director loop
- использовать разные модели для:
  - глубокого reasoning
  - heartbeat/review/policy checks

## 5. Что важно не забыть

- не строить всё вокруг `agent_name`, строить вокруг `run_id`
- не смешивать UI stream и operational bus
- не смешивать память агента, память run и event log
- не пытаться решить race conditions только "аккуратностью кода"
- не превращать директора в хаотичный микроменеджмент без policy и cooldown
- не переносить всю orchestration-логику только в prompt, часть должна быть системной

## 6. Минимальный первый набор правок, с которого стоит начать

Если начинать внедрение прямо сейчас, то первым пакетом надо сделать:

1. `run_id` во всём backend event flow
2. исправление `delegate_parallel`
3. перевод фронтенда на `run_id`
4. потокобезопасную память
5. базовый `RunRegistry`
6. `cancel_sub_agent(run_id)`
7. interruption points в `SubAgent.run()`

Этот пакет уже даст заметный скачок качества и создаст базу под постоянного директора-supervisor.
