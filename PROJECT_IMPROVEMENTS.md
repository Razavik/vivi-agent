# Улучшения проекта

## Краткая оценка

Проект уже имеет хорошую основу: разделение на `agent`, `infra`, `llm`, `safety`, `web`, `client`, наличие диагностических страниц, supervisor-механики, run registry и базового набора тестов. Основная зона роста сейчас — не столько в добавлении новых возможностей, сколько в снижении связности, повышении тестируемости и стабилизации контрактов между компонентами.

## Главные проблемы

### 1. Слишком высокая связность backend-ядра

Наиболее заметно это в `src/web/context.py`, `src/web/routes.py` и частично в `src/app_factory.py`.

Признаки:

- `ServerContext` хранит слишком много ответственности одновременно;
- `Routes` содержит обработку конфигурации, памяти, run lifecycle, артефактов, логов и диагностики;
- изменения в одном месте повышают риск побочных эффектов в другом.

Последствия:

- сложно тестировать изолированно;
- трудно безопасно рефакторить;
- возрастает вероятность регрессий.

### 2. Дублирование логики выполнения агентов

Похожие сценарии присутствуют в:

- `src/agent/runtime.py`;
- `src/agent/sub_agent.py`.

Повторяются:

- загрузка истории и плана из памяти;
- цикл LLM-выполнения;
- обработка ошибок;
- запись снапшотов и итогов сессии;
- отправка событий.

Последствия:

- исправления приходится вносить в несколько мест;
- директор и саб-агенты могут начать вести себя по-разному;
- усложняется добавление новых типов агентов.

### 3. Хрупкость работы с LLM-ответами

`src/llm/ollama_client.py` уже содержит много логики по восстановлению невалидного JSON, чистке markdown-блоков и попыткам починки частично сломанного ответа.

Это полезно, но рискованно:

- поведение трудно предсказать;
- edge-case'ы могут ломать выполнение незаметно;
- без хороших тестов любые изменения опасны.

### 4. Слабая формализация конфигов и persisted state

Сейчас проект опирается на несколько JSON-файлов в `data/`:

- `models.json`;
- `tools.json`;
- `agents.json`;
- `user_profile.json`;
- `runs.json`;
- memory-файлы агентов.

Проблемы:

- нет единой схемы конфигурации;
- часть ошибок просто подавляется fallback-поведением;
- формат хранения не версионируется;
- файловое хранение плохо масштабируется по сложности.

### 5. Недостаточное тестовое покрытие

Существующие тесты полезны, но пока охватывают только часть контрактов. Недостаточно покрыты:

- run lifecycle;
- supervisor-логика;
- обработка плохих ответов модели;
- восстановление состояния после рестарта;
- негативные сценарии и гонки.

### 6. Перегруженный frontend entry point

`client/src/App.tsx` уже объединяет:

- роутинг;
- управление чат-сценарием;
- drag-and-drop изображений;
- confirmation flow;
- supervisor alerts;
- компоновку страниц.

Это не критично прямо сейчас, но со временем будет мешать развивать UI.

## Приоритетные улучшения

### Приоритет 1. Разделить ответственность в backend

Что сделать:

- разбить `Routes` на несколько модулей по областям ответственности;
- сократить обязанности `ServerContext`;
- вынести бизнес-логику из маршрутов в сервисы.

Рекомендуемое целевое разбиение:

- `settings_routes`;
- `memory_routes`;
- `runs_routes`;
- `artifacts_routes`;
- `diagnostics_routes`.

Возможные сервисы:

- `RunService`;
- `MemoryService`;
- `ConfirmationService`;
- `SupervisorService`;
- `ArtifactService`.

Ожидаемый результат:

- меньшая связность;
- более простые unit-тесты;
- более безопасные изменения.

### Приоритет 2. Вынести общий execution loop для агентов

Что сделать:

- вынести общее поведение `AgentRuntime` и `SubAgent` в базовый runtime или отдельный execution engine;
- унифицировать загрузку памяти, обработку step-циклов, event emission и persistence.

Ожидаемый результат:

- меньше дублирования;
- одинаковое поведение директора и саб-агентов;
- проще поддержка и расширение.

### Приоритет 3. Усилить контракты и валидацию

Что сделать:

- формализовать допустимые статусы run;
- централизовать проверку payload для API;
- унифицировать схемы ответов маршрутов;
- ввести явные доменные типы для конфигурации и persisted state.

Ожидаемый результат:

- меньше скрытых ошибок;
- понятнее контракт между backend и frontend;
- проще отлаживать проблемы совместимости.

### Приоритет 4. Усилить диагностику и логирование

Что сделать:

- убрать глухие `except Exception: pass`, где это возможно;
- логировать причины fallback-поведения;
- добавить единые идентификаторы `run_id`, `session_id`, `agent_name` в значимые события;
- фиксировать retries, step latency, причины остановки и причины блокировок.

Ожидаемый результат:

- упрощение отладки;
- лучшее понимание поведения системы;
- более надёжная эксплуатация.

### Приоритет 5. Расширить тесты

Минимальный набор для усиления:

- unit-тесты для `RunRegistry`;
- unit-тесты для `ChatMemoryStore`;
- unit-тесты для `OllamaClient` с набором плохих JSON-ответов;
- интеграционные тесты для маршрутов;
- тесты на `pause`, `resume`, `cancel`, `restore after restart`.

Ожидаемый результат:

- безопасный рефакторинг;
- более предсказуемое поведение;
- меньше регрессий.

### Приоритет 6. Улучшить документацию

`README.md` слишком краткий для проекта такой сложности.

Стоит добавить:

- обзор архитектуры;
- описание ролей директора, саб-агентов и supervisor;
- run lifecycle;
- структуру данных в `data/`;
- список конфигурационных переменных;
- раздел по локальной отладке и troubleshooting.

## Практический roadmap

## Этап 1. Стабилизация контрактов

Цель:

- закрепить доменные инварианты без большого рефакторинга.

Задачи:

- ввести типизированные статусы run;
- проверить все места, где run status читается и записывается;
- унифицировать API-ответы хотя бы для основных маршрутов;
- добавить тесты на критические контракты.

Критерий завершения:

- статусная модель фиксирована;
- ключевые контракты покрыты тестами;
- backend и frontend одинаково понимают состояние run.

## Этап 2. Упрощение backend-композиции

Цель:

- уменьшить связность веб-слоя и orchestration-слоя.

Задачи:

- разрезать `Routes` по зонам ответственности;
- вынести операции из обработчиков в сервисы;
- упростить `ServerContext` до контейнера зависимостей.

Критерий завершения:

- ключевые сервисы можно тестировать независимо от HTTP-слоя;
- маршруты стали тонкими и предсказуемыми.

## Этап 3. Унификация runtime-логики

Цель:

- сократить дублирование между директором и саб-агентами.

Задачи:

- выделить общую execution loop abstraction;
- унифицировать сохранение session state;
- привести event-поток к одному формату.

Критерий завершения:

- поведение runtime-части описывается одним набором правил;
- изменения в общей логике вносятся в одном месте.

## Этап 4. Повышение надёжности хранения состояния

Цель:

- сделать persisted state устойчивее и удобнее для эволюции.

Задачи:

- централизовать чтение и запись конфигов;
- определить схемы JSON-структур;
- подготовить переход части состояния на SQLite при необходимости.

Критерий завершения:

- state layer предсказуем и расширяем;
- снижение риска повреждения или неоднозначной интерпретации данных.

## Самые проблемные файлы для рефакторинга

### `src/web/routes.py`

Нужно:

- уменьшить размер файла;
- убрать смешение транспортного слоя и бизнес-логики;
- формализовать response contract.

### `src/web/context.py`

Нужно:

- сократить число обязанностей;
- вынести orchestration в отдельные сервисы;
- уменьшить количество скрытых side effects.

### `src/agent/runtime.py`

Нужно:

- упростить основной цикл;
- выделить переиспользуемые части;
- улучшить тестируемость step execution.

### `src/llm/ollama_client.py`

Нужно:

- покрыть repair-логику тестами;
- отделить транспортный слой от логики нормализации ответа;
- сделать поведение при ошибках более прозрачным.

## Риски при развитии без рефакторинга

Если продолжать добавлять фичи без структурных улучшений, вероятны следующие проблемы:

- рост количества регрессий;
- усложнение отладки нестабильных сценариев;
- расхождение логики между агентами;
- перегруженность API- и orchestration-слоя;
- замедление разработки из-за высокой связанности.

## Итог

Главная ценность следующих итераций — не просто добавить новые функции, а стабилизировать архитектуру. Самые выгодные вложения сейчас:

- уменьшить связность backend;
- унифицировать runtime-логику;
- усилить тесты и контракты;
- улучшить наблюдаемость;
- подготовить проект к дальнейшему росту без накопления хрупкости.
